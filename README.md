## 🛰️ 2. Propósito de RDM Digital Nexus

**RDM Digital Nexus** es la columna vertebral técnica de **RDM Digital**, Smart City OS turístico‑cultural de Real del Monte:[web:1][web:41][web:59]

- Coordinar la operación turística y cultural del pueblo (rutas, comercios, movilidad, experiencias XR).  
- Ofrecer una capa de identidad soberana para ciudadanos, visitantes y operadores.  
- Orquestar el runtime de IA **Isabella** para análisis, soporte operativo y co‑gobernanza.  
- Unificar decenas de repositorios TAMV en un **hub operativo** con trazabilidad y resiliencia.  

**En términos de MD‑X4**, RDM Digital Nexus se ubica como nodo híbrido entre:

- F1 · DATA (datos turísticos, comerciales, de movilidad).  
- F2 · INTEL (Isabella en contexto territorial).  
- F4 · GOV (reglas de operación, paneles de decisión).  
- F6 · VIS / F7 · TERRITORY (mapas, gemelos digitales, sensores).[web:15][web:41]

## 🏗️ 4. Arquitectura general

### 4.1 Frontend

- **Stack:** React 18 + TypeScript + Vite + Tailwind.  
- **Ejecutor:** Bun para dev/build rápidos.  
- **Ruta operacional principal:**  
  - `/soberano` → panel para ver telemetría de territorio, estado cuántico y sesión soberana en una misma vista.

### 4.2 Backend

- **Stack:** Express + Prisma + Zod.  
- **Seguridad base:** `helmet`, `cors`, `express-rate-limit`, middleware de errores.  
- **Submódulos clave:**
  - `federados` → sincronización GitHub, planes de unificación, grafo multi‑repo.  
  - `isabella` → runtime IA federada, contexto, readiness.  
  - `identity` → registro, desafío, verificación, revocación y auditoría de sesiones/credenciales.  
  - `quantum (L6.Q)` → validación de arquitectura, estado y resiliencia.  
  - `simulation` → escenarios de ataque y scoring de recuperación.  
  - `territory` → estado operativo de mapa, capas de datos, sensores y telemetría turística.

### 4.3 Datos

- Prisma + PostgreSQL para entidades de negocio y tejido social (comercios, rutas, operadores, etc.).  
- Supabase migrations para tablas, índices y políticas RLS.  
- Shield de inmutabilidad en `system_logs` para el bloque génesis `TAMV_GENESIS_ROOT`.

```text
GET  /api/v1/simulation/scenarios

```

**Capacidades de unificación:**

- Descubre y puntúa repos por relevancia (RDM, TAMV, federated‑AI).  
- Construye un **grafo de interconexiones** con `nodes`, `edges`, pesos y razones.  
- Genera un **backlog técnico priorizado** (`P0/P1/P2`) con módulo objetivo e impacto esperado.  
- Construye una cadena circular repo→repo para flujo continuo entre todos los repos activos de un owner, con loopback al origen.  
- Permite fijar `startRepo` (por ejemplo `tamv-digital-nexus`) para arrancar la cadena desde un nodo central.  
- Genera un **plan de consolidación multi‑fase** hacia un `targetRepo` (hub) con secuencia de merge y comandos bootstrap (hasta ~400 repos).  
- Soporta `preset=quantum` para inyectar automáticamente repos base de computación cuántica (TensorFlow Quantum, Microsoft Quantum, QuantumKatas, PennyLane, etc.) al plan de bootstrap.  
- Acepta `seedRepo` (uno o varios) para anexar repos externos estratégicos sin perder prioridad de los repos del owner.

## 🛡️ 8. Seguridad constitucional de tokens

Si defines `FEDERATION_MASTER_KEY`, los JWT emitidos por `/auth/login` y `/auth/signup` incluyen un campo `federationSig` firmado.

- `requireAuth` rechaza tokens sin firma de federación válida.  
- `optionalAuth` adjunta usuario solo cuando la firma es válida, manteniendo compatibilidad para endpoints públicos.  

Sin `FEDERATION_MASTER_KEY`, el sistema opera en modo compatibilidad, sin validación federada pero manteniendo la estructura JWT estándar.

## 🔁 10. CI/CD de unificación real

Workflow principal: `.github/workflows/unification-pipeline.yml`

**Trigger manual (`workflow_dispatch`) con parámetros:**

- `owner`  
- `target_repo`  
- `max_repos`  
- `batch_size`  
- `dry_run`  
- `merge_branch`  

**Ejecutor:** `tools/ci/unification-executor.mjs`

- Obtiene `unification-script` desde la API de RDM Digital Nexus.  
- Ejecuta `git remote add` y `git fetch` por lotes.  
- Ejecuta merges opcionales por rama hacia el `target_repo`.  
- Genera `unification-report.json` con fallos, conflictos y pasos realizados.  

De esta forma, el **hub territorial** se mantiene sincronizado con todo el ecosistema TAMV sin perder trazabilidad.
